## 项目亮点

### 核心痛点：（300 字以内）

解决了哪些业务/用户的技术难题或业务难题

1. **运营商风控收紧，传统外呼/短信触达失灵**：号码被标记、短信被拦截比例升高，触达率与接通率持续下滑，催收/营销/通知场景的 KPI 难以达成。
2. **企业微信/飞书等社交 APP 无开放 API**，企业自有触达系统无法覆盖，业务侧只能依赖人工坐席手动操作，规模受限且合规难控。
3. **人工坐席成本高、效率低、话术不一致**：催收/协商类业务高度依赖人力，单通电话成本高、人均产能受限，且话术质量参差不齐影响合规与转化。

### 关键创新点：（300 字以内）

较行业现有方案的差异及创新

1. **"像人一样操作手机"的通用 RPA 方案**：行业现有方案多依赖厂商 SDK 或短信/外呼通道，JoyPhone 直接走安卓无障碍权限操控真实 APP UI，不依赖任何厂商配合，一套方案覆盖飞书/企微/短信等多通道，绕封禁、抗风控。
2. **端云协同的"手眼脑"分离架构**：手机负责感知（节点树+截图）与执行（点击+输入），云端负责多模态大模型决策与协商话术生成。决策可热更、模型可替换、算力无上限，迭代成本远低于端侧集成方案。
3. **技能库自沉淀形成数据护城河**：每次成功操控的步骤序列被自动固化为"技能"，命中即脚本执行、未命中回退大模型探索。越用越快、越用越准，竞品难以复制业务 know-how 沉淀，形成长期护城河。

## 方案介绍

请详细介绍作品技术/产品方案，可通过架构图等形式辅助介绍，文字内容控制在 500 字以内

JoyPhone 采用 **端云协同架构**：决策在云，感知+执行在端，两端通过 WebSocket 双向实时通信。

**云端（Python / FastAPI）**
- 任务管理：创建/下发协商任务（目标联系人 + 协商目标）
- WebSocket 网关：管理每台手机长连接与单任务会话状态机（NAVIGATING → IN_CHAT → SENT → WAITING_REPLY → NEGOTIATING → DONE/ABORT）
- 决策引擎：输入任务目标+结构化屏幕状态+动作历史，先查技能库匹配；未命中则调多模态大模型推理下一步动作
- 协商机器人：理解对方回复，结合协商目标生成下一轮话术，判断继续/达成/转人工/结束
- LLM 抽象层：统一"多模态输入→结构化输出"接口，便于换模型/加缓存

**安卓端（Kotlin, vivo 定制机, AccessibilityService）**
- 感知模块：读取 AccessibilityNodeInfo 节点树并裁剪（仅保留可见且含文本/可交互节点），按需截图，序列化上传
- 执行模块：将云端动作指令（tap/input/swipe/open_app/back 等）翻译为无障碍 API 调用
- 事件监听：窗口变化/新消息主动上报
- 连接管理：WebSocket 长连 + 断线自动重连

**技能库（雏形）**：将"飞书-给某人发消息"固化为带节点特征的步骤序列；命中即脚本执行，界面对不上则回退大模型探索，探索成功路径再沉淀为新技能。

**关键可测性设计**：真机采集的感知序列存为"回放夹具"，云端可离线回放完整决策闭环，不依赖真机即可在 CI 中反复验证 AI 决策逻辑，是项目质量底座与 TDD 落点。

```
│  任务管理 │ WS网关+会话状态机 │ 决策引擎 │ 协商机器人 │ LLM抽象层 │ 技能库 │
└────────────▲──────────────────────────────┬──────────────────────────────┘
             │ WebSocket（感知↑ / 动作↓）   │
             │  双向实时长连接              │
┌────────────┴──────────────────────────────▼──────────────────────────────┐
│                    安卓端 (Kotlin / AccessibilityService)                 │
│      感知模块（节点树+截图）   │   执行模块（点击/输入/滑动）            │

└───────────────────────────────────────────────────────────────────────────┘
                         ↑ 操控真实 APP（飞书/企微/短信…）
```
