# AI 工作手机 — 飞书协商闭环 MVP 设计文档

- 日期：2026-07-02
- 状态：待评审
- 作者：AI 结对（JoyCode）+ 产品负责人

## 1. 背景与目标

在客服通知、营销推广、逾期催收等批量触达场景中，传统方案受运营商风控收紧影响，号码被标记、短信被拦截比例升高，且无法覆盖企业微信/飞书等社交 APP。企业需要更合规高效的新一代批量触达方案。

技术方向：**由 AI 通过安卓无障碍权限（AccessibilityService）像人一样操控手机上的任意 APP**，模拟人操作逻辑，自动完成消息发送与多轮协商，不依赖任何厂商 SDK。踩过的坑逐步沉淀为可复用「技能（Skill）」，越用越聪明——这是产品护城河。

### 1.1 本 MVP 范围（第一个子项目）

在整个平台级产品中，本 MVP 只聚焦一条**最小端到端闭环**：

> 云端下发协商任务 → 安卓 App 自主打开飞书 → 找到目标联系人 → 发送消息 → 监听对方回复 → 云端大模型进行多轮协商，直到达成目标或转人工。

- 目标 APP：**飞书**（通过无障碍操控，无需飞书配合）
- 决策位置：**云端**（手机负责感知+执行，云端负责决策）
- 通信方式：**WebSocket 长连接**（双向实时）
- 交付形态：**安卓 App（Kotlin） + 云端服务（Python / FastAPI）**

### 1.2 成功标准

云端能驱动一台真实 vivo 定制机，在飞书中对指定联系人完成「发出消息 → 收到回复 → 至少一轮基于回复的协商应答」的完整闭环，并能在异常时安全转人工。

## 2. 关键决策与理由

| 决策点 | 结论 | 理由 |
|---|---|---|
| 是否依赖 vivo SDK | 否，走安卓无障碍 | 现实很可能没有 SDK 提供；无障碍是通用 RPA 方案，可操控任意 APP |
| 决策位置 | 云端 | 易迭代、算力强、便于调 prompt 与换模型 |
| 云端技术栈 | Python (FastAPI)，MVP 阶段 | 瓶颈在大模型推理延迟(I/O)，非语言速度；Python AI 生态最全、迭代最快。后续需将 WS 网关层换 Rust |
| 通信 | WebSocket | 感知上传/动作下发高频，需双向实时 |
| 首攻 APP | 飞书 | 标准 IM，界面规整，适合验证闭环 |
| 技能库 | MVP 仅做「命中即执行」最小版 | 自动沉淀闭环留待后续迭代（YAGNI） |

## 3. 整体架构

端云协同：**决策在云，感知+执行在端**。

### 云端 (FastAPI + Python)
- **任务管理**：创建/下发协商任务
- **WebSocket 网关 + 会话**：管理每台手机连接，维护单次任务的会话状态机
- **决策引擎**：输入=结构化屏幕状态+任务目标+动作历史；输出=下一个动作。调多模态大模型
- **协商文本机器人**：理解对方回复，生成话术，判断继续/达成/转人工/结束
- **LLM 抽象层**：统一封装「多模态输入→结构化输出」，便于换模型/加缓存

### 安卓 App (Kotlin, vivo 定制机)
- **感知模块**：读 AccessibilityNodeInfo 节点树 + 截屏，序列化上传
- **执行模块**：将云端动作指令翻译成无障碍 API 调用
- **事件监听**：窗口变化/通知/飞书新消息主动上报
- **连接管理**：WebSocket 长连 + 断线重连

### 技能库（雏形）
把「飞书-给某人发消息」固化为一串带节点特征的步骤；命中则按脚本执行，界面对不上则回退大模型探索。

## 4. 数据流与 WebSocket 协议

### 4.1 一次协商任务时序
1. 云端创建任务：`{目标联系人, 协商目标}`
2. 云端下发 `task.start` → App
3. App 上报 `perception`（精简节点树 + 按需截图 + 包名/页面）
4. 云端决策引擎下发 `action`（open_app / tap / input / swipe …）
5. App 执行 → 回 `action.result` → 再上报新 `perception`
6. 循环 3–5，直到进入目标会话并发出消息
7. App 监听到对方回复 → 上报 `event.newMessage`
8. 协商机器人生成话术 → 回到发消息动作流；达成目标 → `task.done`

### 4.2 消息格式

上行（App → 云端）：
- `perception`：`{screenshot(base64,可选), nodeTree[], pkg, activity, ts}`
- `action.result`：`{actionId, ok, error?, ts}`
- `event.newMessage`：`{app, sender, text, ts}`
- `heartbeat`：`{deviceId, ts}`

下行（云端 → App）：
- `task.start`：`{taskId, goal, target}`
- `action`：`{actionId, type, params}`，`type ∈ {open_app, tap, input, swipe, back, home, wait, read_screen}`
- `task.done`：`{taskId, result, summary}`
- `task.abort`：`{taskId, reason}`

### 4.3 节点树精简
App 端裁剪：仅保留可见、可交互或含文本的节点，字段精简为 `{id, text, desc, className, bounds, clickable, editable}`。截图按需下发（先给节点树，大模型判断需要看图时才请求截图），降低带宽与 token 成本。

## 5. 决策引擎与协商机器人

### 5.1 决策引擎（UI 操作大脑）
- **输入**：任务目标 + 当前屏幕状态（精简节点树 / 按需截图） + 已执行动作历史
- **处理**：先查技能库是否命中当前任务（如「飞书发消息」）→ 命中则按脚本产出下一步；未命中或界面对不上 → 交多模态大模型推理下一步
- **输出**：结构化 `action`，或 `need_screenshot` / `task.done` / `task.abort`
- **防呆**：单任务最大步数上限、相同屏幕+相同动作重复检测、超时 → abort，避免死循环

### 5.2 协商文本机器人（对话大脑）
- **触发**：收到 `event.newMessage`
- **处理**：结合协商目标 + 对话历史，理解对方意图，生成下一句话术，并判断状态：继续协商 / 已达成 / 需转人工 / 结束
- **输出**：待发送文本（交决策引擎走「输入+发送」动作），或 `task.done` / `task.abort` 信号
- **边界**：MVP 用可配置协商目标 + prompt 模板；敏感或无法处理时转人工，不硬聊

### 5.3 两个大脑的协作 — 会话状态机
`NAVIGATING → IN_CHAT → SENT → WAITING_REPLY → NEGOTIATING → DONE / ABORT`

- 决策引擎负责「手怎么动」（导航到会话、点输入框、发送）
- 协商机器人负责「嘴说什么」（产出要发送的具体文字）
- 会话状态机把两者串联，作为单次任务的唯一事实来源

## 6. 错误处理

- **动作失败**：找不到节点/点击无效 → 重新感知重试（有限次）→ 大模型换策略 → 仍失败则 `task.abort` 转人工
- **意外界面**：弹窗/权限框/登录态失效 → 大模型识别并处理（关弹窗/返回）或 abort
- **死循环防护**：最大步数 + 相同屏幕相同动作重复检测 → 强制中止
- **连接中断**：WS 断线自动重连；任务带状态，重连后可恢复或安全中止
- **回复超时**：等待对方回复设超时窗口，超时后按策略结束或提醒
- **安全兜底**：任何不确定或敏感情形一律转人工，绝不乱操作

## 7. 测试策略

- **云端单测（pytest）**：协议序列化、会话状态机流转、决策引擎「给定屏幕状态→期望动作」（LLM 打桩）、协商机器人话术分支、防死循环逻辑
- **云端集成测试**：用「模拟 App 端」（脚本回放 perception 序列）接 WS，验证完整任务闭环，不依赖真机
- **安卓单测（JUnit）**：节点树裁剪、动作指令→无障碍调用映射、消息编解码
- **真机手测**：飞书实际闭环走查（无障碍行为难自动化，靠手测 + 日志回放）

### 7.1 关键测试基建：可回放的感知序列
把真机采集的 perception 序列存成夹具，云端测试可离线回放，让 AI 决策逻辑在没有真机时也能反复验证。这是本项目最重要的可测性设计，也是 TDD 的落点。

## 8. 范围外（YAGNI，留待后续迭代）

多设备并发调度、技能自动沉淀闭环、除飞书外的其他 APP、可视化运营后台、账号风控对抗、语音外呼。MVP 只证明「云端能驱动一台真机在飞书里完成一轮协商」。
